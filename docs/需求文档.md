## 1. 项目概述

### 1.1 一句话介绍

拾回是一款本地优先的智能相册整理 Android App，帮助用户在手机端快速找回回忆、浏览相册、识别相似照片，并通过保留、删除到回收站、标签归档和压缩入口完成轻量整理。

### 1.2 当前成果定位

当前版本不是上架级产品，而是用于本地验证和阶段演示的高保真 Demo。它已经跑通了以下核心体验：

```text
相册权限授权
-> MediaStore 读取本地照片和视频
-> 首页搜索与照片盲盒
-> 整理页按日期 / 相册 / 相似照片组织
-> 月份详情、日历视图、照片堆叠视图
-> 大图滑动整理、标签归档
-> 删除到 App 回收站
-> 回收站恢复 / 彻底删除确认
-> 压缩选择页
-> Debug APK 构建与真机安装测试
```

### 1.3 产品关键词

- 本地优先
- 回忆找回
- 相册减负
- 快速整理
- 安全删除
- 智能搜索
- 纸纹拟物视觉
- 真机可验证 Demo

## 2. 用户问题与设计目标

### 2.1 用户问题

手机相册长期累积后会出现几个典型问题：

- 照片、截图、视频混在一起，用户很难快速定位某段回忆。
- 系统相册适合浏览，但不适合批量整理和决策。
- 相似照片、连拍照片、截图占用空间，但用户不敢直接删除。
- 用户希望用自然关键词搜索内容，例如“猫”“风景”“合照”“某个地点”。
- 删除照片属于高风险动作，需要可恢复、可确认的安全流程。

### 2.2 当前版本设计目标

- 做出一个能安装到 Android 手机上的完整交互 Demo。
- 用真实系统媒体数据驱动页面，而不是只做静态 UI。
- 让首页从“功能入口”变成“找回回忆”的情绪化入口。
- 让整理页承担日期、相册、相似照片三种主要整理路径。
- 用大图滑动整理降低单张照片决策成本。
- 先用 App 内回收站承接删除动作，避免误删焦虑。
- 在本地完成搜索索引、内容标签、地点和人脸检测，不上传照片。

### 2.3 当前版本不做

- 不做账号登录、云同步、会员付费。
- 不做上架发布和生产级隐私合规流程。
- 不做真实视频压缩编码。
- 不做完整系统级永久删除闭环，当前以 App 内模拟删除为主。
- 不做同一人物身份聚类命名，当前仅检测“有人脸/人物”。
- 不做 Room 持久化整理状态，当前整理状态主要保存在 ViewModel 内存中。

## 3. 当前产品架构

底部一级导航为三个模块：

```text
首页 | 整理 | 我的
```

二级页面包括：

```text
回收站
压缩
月份详情
相册详情
相似照片组详情
大图整理页
```

### 3.1 首页

首页定位为“找回回忆”的入口，强调情绪化浏览和智能搜索。

当前已实现：

- 纸纹背景和浅色拟物视觉。
- 顶部欢迎语。
- 右上角消息样式圆形按钮。
- 搜索框，可搜索照片内容。
- 相册标签横向快捷入口。
- 照片盲盒，展示中间主图和左右侧图。
- 盲盒支持刷新、保留、删除到回收站、添加标签。
- 照片概览模块，展示相似照片、截图、视频等整理入口。
- 自定义底部 Tab 栏。

搜索能力覆盖：

- 文件名。
- 相册名。
- 日期。
- 媒体类型，例如照片、视频、截图。
- 用户添加的标签。
- EXIF 地点信息和反向地理编码结果。
- ML Kit 图片标签，例如风景、宠物、猫、狗、食物、天空、建筑。
- ML Kit 人脸检测结果，例如人物、人像、自拍、合照。

### 3.2 整理

整理页是核心工作台，当前包含三种模式：

```text
日期 | 相册 | 相似
```

当前已实现：

- 顶部整理进度，显示已整理数量、总数量和百分比。
- 白色圆角进度条。
- 液态滑块式分段控制。
- 日期、相册、相似三模式切换。
- 右侧箭头按钮可顺序切换模式。
- 日期模式按年份和月份组织。
- 月份卡片显示照片堆叠、月份、照片/视频数量、已整理数量、总大小和进度环。
- 相册模式按系统 bucket 分组，展示相册封面、名称、数量和大小。
- 相似模式展示本地识别出的相似照片组，推荐保留项用星标标记。

### 3.3 月份详情

从日期卡片进入月份详情。

当前已实现：

- 顶部返回。
- 月份下拉选择。
- 日历视图。
- 照片堆叠视图。
- 日历中有照片的日期展示缩略图。
- 无照片日期展示灰色斜杠占位。
- 支持在日历和堆叠视图之间切换。
- 支持选择媒体后进入压缩或删除流程。

### 3.4 相册详情

从首页相册标签或整理页相册卡片进入。

当前已实现：

- 按系统相册分组。
- 使用真实媒体封面。
- 相册详情中支持媒体网格浏览。
- 长按或选择后出现删除、压缩等操作入口。

### 3.5 相似照片

当前相似照片不是静态假数据，而是本地后台识别结果。

当前已实现：

- 对照片生成 dHash。
- 用汉明距离判断视觉相似度。
- 结合同日期、同相册、连拍时间窗口提高聚类可信度。
- 每组选择一个推荐保留项。
- 推荐保留项在 UI 中显示黄色星标。
- 点击相似照片组可进入只包含当前组的大图整理页。
- 整理完当前组后返回相似照片列表。

### 3.6 大图整理页

大图整理页用于快速处理单张照片。

当前已实现：

- 大图居中展示。
- 顶部返回与更多菜单。
- 更多菜单包含添加相册、切换手势。
- 底部缩略图列表，可快速切换当前照片。
- 保留和删除按钮。
- 支持滑动手势整理。
- 支持向下滑动返回照片库。
- 滑动时出现视觉反馈：保留为绿色提示，删除为红色提示。
- 支持标签归档，标签包括收藏、家庭、旅行、工作、待整理等。
- 我的页可切换手势习惯。

当前手势模式：

```text
左右滑动：左滑保留，右滑删除
上滑删除：上滑删除，左滑保留
```

### 3.7 回收站

回收站承接所有删除动作，降低误删风险。

当前已实现：

- 从“我的”页进入回收站。
- 回收站网格展示已删除到 App 内的媒体。
- 显示媒体数量和文件大小。
- 支持选择态。
- 支持恢复。
- 支持彻底删除。
- 支持全部删除。
- 彻底删除前有二次确认弹窗。

当前删除边界：

- 删除到回收站：已实现，App 内隐藏普通列表中的对应媒体。
- 彻底删除：当前将媒体从 App 回收站和 App 可见列表中移除，属于 Demo 级模拟永久删除。
- 系统相册真实删除：已预留 `SystemMediaDeleteHelper`，但当前主流程未作为生产能力启用。

### 3.8 压缩页

压缩作为整理流程中的二级页面，不作为底部 Tab。

当前已实现：

- 从选择状态进入压缩页。
- 顶部显示待压缩数量和预计节省空间。
- 网格显示待压缩媒体。
- 每项显示缩略图和文件大小。
- 底部压缩设置面板。
- 质量选项：低、中、高。
- 开始压缩按钮。

当前压缩边界：

- 压缩页 UI 和选择流程已完成。
- 当前点击开始压缩后返回整理页，未实现真实图片/视频编码压缩。

### 3.9 我的

我的页用于设置、状态和安全入口。

当前已实现：

- 回收站入口。
- 主题切换卡片。
- 手势习惯设置。
- 每组照片数量设置：15、30、60、无限。
- 关于 App。
- 权限状态。
- 选择更多权限入口。
- 清空本地整理记录入口。

## 4. 关键用户流程

### 4.1 首次授权流程

```text
打开 App
-> 权限说明页
-> 请求相册权限
-> Android 13+ 请求 READ_MEDIA_IMAGES / READ_MEDIA_VIDEO
-> Android 14+ 兼容 READ_MEDIA_VISUAL_USER_SELECTED
-> 判断 None / Partial / Full
-> 进入首页并读取可访问媒体
```

### 4.2 首页找回回忆流程

```text
进入首页
-> 搜索关键词或查看照片盲盒
-> 浏览推荐照片
-> 保留 / 删除 / 添加标签
-> 通过相册标签跳转整理页对应相册
```

### 4.3 日期整理流程

```text
进入整理页
-> 选择日期模式
-> 查看年份和月份卡片
-> 点击某个月份
-> 在日历视图或照片堆叠视图中浏览
-> 选择照片
-> 删除到回收站或进入压缩页
```

### 4.4 相似照片整理流程

```text
进入整理页
-> 选择相似模式
-> 后台完成相似照片识别
-> 查看相似照片组
-> 进入某一组的大图整理页
-> 参考星标推荐保留项
-> 滑动保留或删除
-> 返回相似照片列表
```

### 4.5 安全删除流程

```text
用户点击删除或滑动删除
-> 媒体加入 App 回收站
-> 从普通列表隐藏
-> 回收站中可恢复
-> 彻底删除前弹出二次确认
-> Demo 中从 App 状态移除
```

## 6. 技术路线

### 6.1 技术栈

```text
语言：Kotlin
UI：Jetpack Compose + Material 3
架构：MVVM + Repository
状态管理：ViewModel + StateFlow
媒体读取：MediaStore
图片加载：Coil Compose
本地智能：ML Kit Image Labeling + Face Detection
相似识别：本地 dHash + 汉明距离
地点识别：ExifInterface + Geocoder
本地缓存：SharedPreferences + JSON
构建工具：Gradle Wrapper
真机调试：ADB
```

### 6.2 架构说明

当前项目采用单 Activity + Compose 页面组合。

```text
MainActivity
-> 权限请求、页面路由、底部导航

PhotoLibraryViewModel
-> 媒体状态、选择状态、整理状态、回收站状态、搜索索引状态

MediaStorePhotoRepository
-> 从系统 MediaStore 查询照片和视频

PhotoContentAnalyzer
-> 调用 ML Kit 生成图片标签和人脸数量

PhotoLocationExtractor
-> 读取 EXIF GPS 并反向地理编码

SimilarPhotoDetector
-> 生成 dHash 并聚类相似照片

SearchIndexStore
-> 缓存内容标签、地点索引、人物命名结构

UI Screens
-> Home / Tidy / Mine / RecycleBin / Compress
```

### 6.3 权限策略

```text
Android 13 及以上：
READ_MEDIA_IMAGES
READ_MEDIA_VIDEO

Android 14 及以上：
READ_MEDIA_VISUAL_USER_SELECTED

Android 12 及以下：
READ_EXTERNAL_STORAGE

读取原图 GPS：
ACCESS_MEDIA_LOCATION
```

访问状态分为：

```text
None：未授权
Partial：仅授权部分照片
Full：授权全部照片和视频
```

### 6.4 媒体读取策略

- 通过 MediaStore 读取图片和视频。
- 当前最多读取最近 500 项。
- 优先按 `DATE_TAKEN` 倒序，缺失时使用 `DATE_ADDED`。
- 不直接扫描文件路径。
- UI 中使用内容 Uri 和 Coil 加载缩略图。
- 图片内容分析前先降采样，避免大图解码导致卡顿或内存压力。

### 6.5 本地智能策略

当前智能能力全部在本机执行：

- ML Kit 图片标签用于搜索语义补全。
- ML Kit 人脸检测用于识别人物、人像、自拍、合照。
- EXIF GPS 用于地点搜索。
- dHash 用于相似照片识别。
- 搜索索引缓存到本地，避免每次启动重复分析。

## 8. 项目工作流总结

### 8.1 需求到成果的工作流

```text
1. 明确产品方向
   从“智能相册整理”收敛到“本地优先、找回回忆、安全整理”。

2. 编写第一版 PRD
   先定义三 Tab、权限、读取相册、整理、回收站、压缩等核心模块。

3. 搭建 Android 原生工程
   使用 Kotlin + Jetpack Compose 创建可构建、可安装的基础 App。

4. 跑通真实数据链路
   接入权限请求和 MediaStore，保证页面使用真实手机照片和视频。

5. 按参考图和 Figma 迭代 UI
   首页、整理页、日期页、相册页、回收站、压缩页、大图整理页多轮截图对照调整。

6. 增加智能能力
   加入 ML Kit 标签、人脸检测、EXIF 地点、dHash 相似照片识别和搜索缓存。

7. 真机验证
   通过 Gradle 构建 APK，用 ADB 安装到 Android 真机，截图检查页面和交互。

8. 复盘回写文档
   将 PRD 从“计划文档”改成“当前成果文档”，用于阶段复盘和对外说明。
```

### 8.2 当前迭代方式

```text
需求描述 / 参考图
-> 拆解页面和交互
-> 修改 Compose 代码
-> Gradle 构建
-> ADB 安装
-> 真机截图
-> 对照参考图调整间距、字号、颜色、动效
```

## 9. 主要问题与解决方法

### 9.1 中文路径导致 Android 构建不稳定

问题：

- 初始资料放在中文路径下，Android Gradle Plugin、Gradle 缓存和部分工具链在非 ASCII 路径下更容易出现兼容问题。

解决：

- 将代码工程放到英文路径 `C:\AndroidProjects\PhotoTidyAndroid`。
- 文档、截图和参考资料继续保留在桌面中文资料目录。

### 9.2 Android 新版相册权限复杂

问题：

- Android 13 引入图片/视频分离权限。
- Android 14 进一步引入“部分照片授权”。
- 用户可能只授权部分照片，不能当作异常处理。

解决：

- 设计 `PhotoAccessStatus`：None、Partial、Full。
- 根据系统版本请求不同权限。
- Partial 状态下仍读取 MediaStore 返回的可访问媒体。
- 我的页提供“选择更多”入口。

### 9.3 大量照片读取和分析可能卡顿

问题：

- 手机相册可能有上千张照片。
- 如果一次性解码原图或全量分析，会造成卡顿、内存压力和耗电。

解决：

- 当前读取最近 500 项作为 Demo 范围。
- MediaStore 只取必要字段和内容 Uri。
- Coil 负责 UI 缩略图加载。
- 内容分析前降采样。
- 后台任务使用协程 IO 线程。
- 搜索索引缓存后避免重复扫描。

### 9.4 相似照片不能依赖云端 AI

问题：

- 上传照片做相似识别不符合本地优先和隐私定位。
- Demo 阶段也不适合接重型模型。

解决：

- 使用本地 dHash 生成感知哈希。
- 通过汉明距离判断视觉相似。
- 叠加同日期、同相册、连拍时间窗口提升准确性。
- 用分辨率、文件大小、是否截图等规则推荐保留项。

### 9.5 搜索只按文件名不够智能

问题：

- 用户更可能搜索“猫”“食物”“合照”“风景”，而不是文件名。

解决：

- 建立综合搜索索引。
- 将文件名、相册、日期、类型、标签、地点、ML Kit 标签、人脸信息合并成搜索文本。
- 对英文标签增加中文别名，例如 landscape 对应风景、自然、户外。
- 将内容分析结果缓存到本地。

### 9.6 删除照片有信任风险

问题：

- 真正删除系统相册照片是高风险操作。
- Android 11+ 系统删除需要用户确认，不能静默删除。

解决：

- 当前版本先做 App 内回收站。
- 删除只是从普通列表隐藏并进入回收站。
- 回收站支持恢复。
- 彻底删除前二次确认。
- 真实系统删除 helper 已预留，待流程稳定后再接入主流程。

### 9.7 UI 细节需要和参考图贴近

问题：

- 阶段 Demo 不仅要“能用”，还要能展示视觉完成度。
- Compose 默认控件很容易显得普通，需要大量定制。

解决：

- 采用纸纹背景、白色半透明面板、橙色强调色、圆角卡片和照片堆叠。
- 自定义首页底部栏、液态分段控件、月份卡片、相册卡片、回收站网格。
- 每轮改动后真机截图，对照参考图继续调整边距、字号、遮挡和裁切问题。

### 9.8 Room / KSP 暂不进入主路径

问题：

- 当前模板使用较新的 AGP 和 Kotlin，Kapt / KSP 兼容需要额外处理。
- Demo 阶段尚未真正需要复杂数据库表。

解决：

- 当前整理状态先放在 ViewModel。
- 搜索索引用 SharedPreferences + JSON 轻量持久化。
- 将 Room runtime 作为依赖保留，等整理状态持久化阶段再接入 compiler。
