## 1. 当前技术目标

当前版本的目标不是上架级产品，而是完成一个可验证、可展示、可复盘的作品集项目：

```text
权限授权
-> 读取真实系统媒体
-> 首页回忆与智能搜索
-> 日期 / 相册 / 相似照片整理
-> 大图滑动决策
-> App 内回收站
-> 压缩选择页
-> 真机安装和截图验证
```

核心技术原则：

- 原生 Android 优先。
- 本地处理优先。
- UI 用 Jetpack Compose，不使用 XML layout 写界面。
- 高风险能力先模拟，流程稳定后再接系统级删除和真实压缩。
- 所有视觉迭代以真机截图为准。

## 4. 权限路线

当前目标系统覆盖 Android 12 及以下、Android 13、Android 14+。

```text
Android 13+：
READ_MEDIA_IMAGES
READ_MEDIA_VIDEO

Android 14+：
READ_MEDIA_VISUAL_USER_SELECTED

Android 12 及以下：
READ_EXTERNAL_STORAGE

读取原图 GPS：
ACCESS_MEDIA_LOCATION
```

权限状态：

```text
None：未授权
Partial：仅授权部分照片
Full：授权全部照片和视频
```

处理策略：

- 未授权时显示权限说明页。
- Partial 不视为错误，MediaStore 会返回用户已授权的媒体。
- Full 状态读取图片和视频。
- Partial 状态优先保证图片读取和首页可用。
- 我的页提供“选择更多”入口。

读取策略：

## 6. 搜索与本地智能路线

### 6.1 搜索索引来源

首页搜索会综合以下信息：

```text
文件名
相册名
日期
照片 / 视频 / 截图类型
用户标签
地点字段
ML Kit 图片标签
ML Kit 人脸检测结果
人物命名字段
```

### 6.2 ML Kit 图片标签

文件：

```text
PhotoContentAnalyzer.kt
```

策略：

- 使用 ML Kit Image Labeling。
- 只分析照片。
- 默认批量最多分析 40 张新增照片。
- 分析前将图片降采样到目标尺寸。
- 置信度阈值为 0.62。
- 英文标签映射中文别名，例如风景、猫、狗、食物、建筑、天空等。

### 6.3 ML Kit 人脸检测

策略：

- 使用 ML Kit Face Detection。
- 采用 FAST 模式。
- 检出人脸后加入“人物、人像、人脸、自拍、合照”等搜索词。
- 当前只做有人脸检测，不做同一人物身份聚类。

### 6.4 地点索引

文件：

```text
PhotoLocationExtractor.kt
```

策略：

- 使用 `ExifInterface` 读取 GPS。
- Android Q+ 优先通过 `MediaStore.setRequireOriginal(uri)` 读取原图信息。
- 使用 `Geocoder` 反向地理编码。
- 缓存已检查但无 GPS 的照片，避免每次启动重复扫描。

### 6.5 搜索缓存

文件：

```text
SearchIndexStore.kt
```

策略：

- 使用 SharedPreferences 存 JSON。
- 为每张照片保存签名：id、大小、时间、宽高、mimeType。
- 媒体签名变化时丢弃旧缓存。
- 缓存内容标签、地点索引、人物命名结构。

## 7. 相似照片路线

文件：

```text
SimilarPhotoDetector.kt
```

当前算法：

```text
1. 读取照片缩略图并降采样
2. 生成 9x8 dHash
3. 使用汉明距离比较相似度
4. 结合日期、相册、连拍时间窗口判断相似组
5. 每组按推荐分数选出最佳保留项
```

相似判定：

```text
严格距离：<= 4
同上下文距离：<= 8 且同日期或连拍时间接近
宽松距离：<= 10 且同相册并同日期或连拍时间接近
连拍窗口：10 分钟
```

推荐保留分数：

```text
分辨率分数
+ 文件大小权重
+ 拍摄时间
- 截图惩罚
```

当前边界：

- 全部在本地执行。
- 不上传照片。
- 不显示 AI 推荐理由，只显示星标。
- 误判需要用户最终决策。

## 8. 删除与回收站路线

当前删除策略：

```text
普通删除
-> 加入 App 内 recycleBinIds
-> 从 visibleMedia 中隐藏
-> 回收站可恢复

彻底删除
-> 二次确认
-> 从 recycleBinIds 移到 permanentlyDeletedIds
-> Demo 中从 App 可见状态移除
```

为什么先不直接删系统相册：

- 真删除是高风险动作。
- Android 11+ 删除媒体必须通过系统确认请求。
- 不同厂商系统行为存在差异。
- 作品集 Demo 先证明安全流程和交互闭环。

已预留：

```text
SystemMediaDeleteHelper.kt
```

后续可以通过 `MediaStore.createDeleteRequest` 接入系统级删除确认。

## 11. 主要问题与解决方法

### 11.1 中文路径构建风险

问题：

- Android 工程在中文路径下更容易遇到 Gradle、AGP、缓存路径兼容问题。

解决：

- 代码工程固定放在 `C:\AndroidProjects\PhotoTidyAndroid`。
- 桌面中文目录只放文档和参考资料。

### 11.2 Android 14+ 部分授权

问题：

- 用户可能只授权部分照片，App 不能假定拥有完整媒体库。

解决：

- 建立 `PhotoAccessStatus`。
- Partial 状态也进入 App。
- UI 中提供选择更多入口。

### 11.3 大量媒体导致性能压力

问题：

- 一次性读取、解码、分析全部照片会卡顿。

解决：

- Demo 阶段限制最近 500 项。
- 只存内容 Uri。
- Coil 负责 UI 加载。
- 分析前降采样。
- 后台协程执行。
- 缓存搜索索引。

### 11.4 搜索语义不足

问题：

- 文件名搜索不能满足用户找回回忆的需求。

解决：

- 合并文件名、相册、日期、类型、标签、地点、ML Kit 标签、人脸词。
- 建立中文别名映射。
- 用本地缓存减少重复分析成本。

### 11.5 真删除风险

问题：

- 真删除照片不可轻易上线到 Demo 主流程。

解决：

- 先使用 App 内回收站。
- 彻底删除前二次确认。
- 系统删除能力先预留，不默认启用。

### 11.6 视觉还原难度

问题：

- 参考图中的纸纹背景、拟物照片卡、液态控件、堆叠效果用默认控件难以实现。

解决：

- 自定义 Compose 组件。
- 反复真机截图检查。
- 针对遮挡、裁切、字号、间距逐项调整。
